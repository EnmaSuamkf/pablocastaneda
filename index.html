free-code

The first agent designed and built for development — flexible and open source.

The first agent to natively ship tools that accelerate development and bring value to devs and companies every day. Designed to accelerate every stage of a project: it doesn't just focus on understanding the codebase, writing and testing code — it's also flexible for organizations, with the ability for every company to modify it to fit their needs.

  • Any model — Anthropic, Google Vertex, or open-source models like Qwen Coder, Llama, and DeepSeek via LM Studio, Ollama, or any OpenAI-compatible endpoint
  • Vision Mode — free-code is the first agent with augmented reality: it can understand what you say and respond back. It can help you understand your surroundings — for example, taking screenshots of what you're looking at and giving you solutions or advice, something no other agent is capable of today.
  • Code graph — as you already know, context is everything. With native code-graph, free-code can generate code graphs of the project and understand it far better, in less time and spending fewer tokens. This takes the agent to the next level.
  • Agent browser — navigates the web natively over CDP: browsing, searching, and fetching whatever the task needs
  • Subagents — parallel agent sessions, each with its own context and tools, working on independent subtasks
  • Local RAG — today an agent needs more information in a secure and reliable way; with its local RAG, free-code can manage private knowledge bases, letting the agent extend its knowledge without retraining a model.
  • Skills & MCP — reusable behaviors, slash commands, and any MCP server plugged in as new tools
  • Webhooks — free-code is the first open-source agent able to receive external events as webhooks, something unthinkable not long ago.

Open Source · Automation

agent-webhook-bridge

Until today, Claude could easily communicate with the outside world through MCPs or by making curl calls. But it couldn't receive events from the outside — that has changed with agent-webhook-bridge (AWB).

AWB gives Claude (I'm working on adding Cursor, Codex, and Copilot) an inbound door: an external event — a GitHub push, a CI failure, a Flowise flow, a plain curl — wakes the agent, with no polling and no need for the agent to be running beforehand. All of it secure: nobody can trigger it without your secret.

  • HTTP-triggered — one POST spawns the agent headless, as a fresh session or resuming a previous conversation
  • Secure by default — nothing is exposed unless you decide to; every hook gets its own secret, without it the webhook won't fire
  • Configurable per hook — --workdir, --prompt-template, --permission-mode, --trigger/--queue, --consumer, --visible: each endpoint defines how its agent runs
  • Event protocol — events fire the agent the moment they arrive, or persist in SQLite as a queue for later consumption

agent-mesh

A hub where operators publish AI agents and anyone can send them tasks from a simple web UI.

The first working piece of the AgentMesh vision: operators publish AI agents running on their own machines, and anyone can send them tasks from a web UI. The hub never runs agents itself — it routes jobs and collects results, turning idle agent capacity into a shareable service.

  • Complete task loop — submit → run → result, with agents spawned headless in sandboxed workdirs
  • Session chaining — jobs can resume previous Claude sessions, so conversations continue across tasks
  • Remote access — shareable through a cloudflared tunnel, with per-user API keys that expire, rate-limit, and cap usage
  • Zero dependencies — a single-process Node hub: no build step, nothing to install

Open Source · AI Orchestration

target

A local hub that runs multi-step Claude Code workflows as one continuous, self-checking conversation.

Instead of a registry of loose, parallel jobs, target creates one dedicated agent per workflow: its steps run strictly in order, always resuming the same Claude session, so the whole workflow reads as a single conversation that advances step by step — with each step able to judge its own result before moving on.

  • Chained sessions — every step in a workflow resumes the previous Claude session, keeping full context from start to finish
  • Subagent delegation — each step is dispatched to a Task-tool subagent, so intermediate tool calls never bloat the resumed session
  • Judge-driven retries — steps with an acceptance criterion get a self-evaluation pass, and are automatically retried with feedback until it passes or retries run out
  • On-demand reruns — any step can be re-run in isolation on a fresh session, with the workflow's status reconciled from its steps afterward
  • Per-step run selection — Start/Resume/Restart each accept a subset of step ids, exposed in the UI as checkboxes

agent-webhook-bridge

Hasta hoy en día Claude podía fácilmente comunicarse con el mundo exterior a través de MCPs o haciendo llamadas curl. Pero no podía recibir eventos desde afuera; esto ha cambiado con agent-webhook-bridge (AWB).

AWB le da a Claude (estoy trabajando para agregar Cursor, Codex y Copilot) una puerta de entrada: un evento externo — un push de GitHub, un fallo de CI, un flujo de Flowise, un curl simple — despierta al agente, sin polling y sin necesidad de que el agente esté corriendo de antemano. Todo de forma segura: nadie puede dispararlo sin tu secreto.

  • Activado por HTTP — un solo POST lanza al agente en modo headless, como una sesión nueva o retomando una conversación anterior
  • Seguro por defecto — nada queda expuesto a menos que tú lo decidas; cada hook tiene su propio secreto, sin él el webhook no se dispara
  • Configurable por hook — --workdir, --prompt-template, --permission-mode, --trigger/--queue, --consumer, --visible: cada endpoint define cómo se ejecuta su agente
  • Protocolo de eventos — los eventos disparan al agente en el momento en que llegan, o se persisten en SQLite como una cola para consumo posterior

agent-mesh

Un hub donde los operadores publican agentes de IA y cualquiera puede enviarles tareas desde una interfaz web simple.

La primera pieza funcional de la visión de AgentMesh: los operadores publican agentes de IA que corren en sus propias máquinas, y cualquiera puede enviarles tareas desde una interfaz web. El hub nunca ejecuta agentes por sí mismo — enruta trabajos y recopila resultados, convirtiendo la capacidad ociosa de agentes en un servicio compartible.

  • Ciclo de tareas completo — enviar → ejecutar → resultado, con agentes lanzados en modo headless en workdirs aislados
  • Encadenamiento de sesiones — los trabajos pueden retomar sesiones anteriores de Claude, así las conversaciones continúan a través de las tareas
  • Acceso remoto — compartible mediante un túnel de cloudflared, con claves de API por usuario que expiran, limitan la tasa y acotan el uso
  • Cero dependencias — un hub de Node de un solo proceso: sin paso de compilación, nada que instalar

Código abierto · Orquestación de IA

target

Un hub local que ejecuta flujos de trabajo de Claude Code de varios pasos como una única conversación continua y autoverificada.

En lugar de un registro de trabajos sueltos y paralelos, target crea un agente dedicado por flujo de trabajo: sus pasos se ejecutan estrictamente en orden, retomando siempre la misma sesión de Claude, de modo que todo el flujo de trabajo se lee como una única conversación que avanza paso a paso — con cada paso capaz de evaluar su propio resultado antes de continuar.

  • Sesiones encadenadas — cada paso de un flujo de trabajo retoma la sesión anterior de Claude, manteniendo el contexto completo de principio a fin
  • Delegación en subagentes — cada paso se despacha a un subagente de la herramienta Task, así las llamadas a herramientas intermedias nunca inflan la sesión retomada
  • Reintentos guiados por un juez — los pasos con un criterio de aceptación reciben una pasada de autoevaluación, y se reintentan automáticamente con retroalimentación hasta que pasan o se agotan los reintentos
  • Reejecuciones bajo demanda — cualquier paso puede volver a ejecutarse de forma aislada en una sesión nueva, y el estado del flujo de trabajo se reconcilia después a partir de sus pasos
  • Selección de ejecución por paso — Start/Resume/Restart aceptan cada uno un subconjunto de ids de pasos, expuesto en la interfaz como casillas de verificación
